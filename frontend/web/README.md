# Web frontend — BSBA Freshman Scheduling Tool

Static HTML/CSS/JS app (no build step, no framework) built from
`../bsba-scheduling-tool-mockup.html`. Primary data source is
`schedule_engine`'s generated artifacts — real, conflict-checked, per-major
cohort schedule blocks (see `../../schedule_engine/README.md`) — not the
older pooled `mining/co_occurrence.py` output.

## Run it locally, no AWS deploy needed

Browsers block `fetch()` of local JSON over `file://`, so open this folder
through a tiny static server instead of double-clicking `index.html`:

```
cd frontend/web
python -m http.server 8000
```

Then open **http://localhost:8000**. The Major picker loads one of the
files in `data/artifacts/` (bundled copies of the repo-root `artifacts/`
directory) — each is a specific major's set of 4–8 generated schedule
blocks. Everything renders straight from that JSON: no server, no AWS,
no build step. The Block picker (or "View on calendar" in the comparison
list) swaps which block is shown in the top card, the week-view calendar,
and the rationale text.

Any other static server works the same way — `npx serve`, VS Code's Live
Server extension, etc. The only requirement is "served over http, not
opened as a file."

## What's real vs. computed client-side

- **Blocks, courses, sections, times, seat counts, advisories** — straight
  from the artifact JSON, generated by `schedule_engine` against the real
  Fall 2026 section catalog. Nothing here is invented by the frontend.
- **Fit score** — the generator's beam-search ranking number, shown with
  its definition attached everywhere it appears (see `SCORE_DEFINITION` in
  `app.js`). It's a within-major ranking signal, not a percentage.
- **"Why this block"** — computed client-side from the block's own fields
  (units, campus days, most-popular course, advisory/TBA counts). No LLM
  call, no network dependency, no error state to handle.
- **Calendar** — only plots courses with a real `days`/`start`/`end` on
  the section. Anything else (TBA face-to-face sections, async online
  sections) is excluded from the grid and explicitly listed underneath it
  instead of being silently dropped.

## The Q&A box is separate and legacy

"Ask a question" at the bottom still calls the older `/ask` API
(`api/handlers/ask.py` → `bedrock/client.py`), grounded in the pooled,
all-concentration `mining/co_occurrence.py` output — general historical
background, not specific to whatever block/major is currently on screen.
It needs a deployed API to do anything (see below); with `API_BASE_URL`
left blank it says so rather than pretending to answer.

## Point Q&A at the real AWS API

Once `infra/deploy.sh` has run and you have the `ApiUrl` stack output, set
`API_BASE_URL` in `config.js` to that URL (no trailing slash needed). The
status ribbon at the top reflects whether Q&A is live or offline. There is
no deployed endpoint for the schedule_engine artifacts themselves yet —
that's still a CLI-only pipeline (see `schedule_engine/README.md`); the
Major/Block pickers always read the bundled local JSON regardless of
`API_BASE_URL`.

## Refreshing the bundled artifact data

`data/artifacts/*.json` are point-in-time copies of the repo-root
`artifacts/*.json`. Regenerate a major with:

```
python3 -m schedule_engine generate --major "BA - Marketing"
```

then copy it over:

```
cp ../../artifacts/ba_marketing.json data/artifacts/ba_marketing.json
```

## Editing the major list

`config.js`'s `MAJOR_ARTIFACTS` array is the picker's source of truth —
add/remove a filename there (and make sure the matching file exists in
`data/artifacts/`) to change what shows up.

## Deploying as a real static site

Any static host works (S3 + CloudFront, GitHub Pages, Netlify). Upload
`index.html`, `styles.css`, `config.js`, `app.js`, and `data/` as-is — set
`API_BASE_URL` in `config.js` first if you want live Q&A.
